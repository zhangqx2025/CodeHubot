# 文档切分方式说明

## 📋 概述

文档向量化的关键步骤之一是将长文档切分成合适大小的文本块。不同的切分方式会影响检索效果和响应质量。本系统提供了4种切分方式供您选择。

---

## 🎯 切分方式对比

| 切分方式 | 特点 | 优势 | 适用场景 | 推荐指数 |
|----------|------|------|----------|----------|
| **固定大小** | 按固定字符数切分，在句子边界处断开 | 块大小均匀，检索效率高 | 大部分文档 | ⭐⭐⭐⭐⭐ |
| **按段落（双换行）** | 以双换行符为分隔符，保持段落完整性 | 语义连贯性好 | 结构化文档、教材 | ⭐⭐⭐⭐ |
| **按句子** | 在句子边界处切分 | 适合问答场景 | FAQ、短文本 | ⭐⭐⭐ |
| **自定义** | 手动设置块大小和重叠 | 灵活度最高 | 特殊需求 | ⭐⭐ |

---

## 📖 详细说明

### 1️⃣ 固定大小切分（推荐）

**原理**：
- 按照固定的字符数（默认500字符）切分文本
- 在切分点附近寻找句子边界（句号、问号、叹号、换行符）
- 保留一定的重叠区域（默认50字符），增强上下文连贯性

**参数**：
- 块大小：500 字符
- 重叠大小：50 字符

**示例**：
```
原文：
这是第一句话。这是第二句话。这是第三句话。这是第四句话...（共1000字符）

切分结果：
块1 [0-500]: 这是第一句话。这是第二句话...（在句号处结束）
块2 [450-950]: ...第二句话。这是第三句话...（包含50字符重叠）
块3 [900-1000]: ...这是第四句话。
```

**优点**：
- ✅ 块大小均匀，便于向量化和检索
- ✅ 在句子边界切分，语义完整
- ✅ 重叠区域保证上下文连贯性
- ✅ 适用性最广

**缺点**：
- ❌ 可能在段落中间切分

**适用场景**：
- 📄 论文、报告
- 📚 教学资料
- 📝 技术文档
- 💬 对话记录

---

### 2️⃣ 按段落切分（双换行）

**原理**：
- **以双换行符（`\n\n`）为段落分隔符** ⭐重点
- 单个换行（`\n`）不会切分，保留在段落内
- 将多个段落合并成一个文本块，直到接近块大小限制
- 如果段落本身超过块大小，再按固定大小切分

**参数**：
- 块大小限制：500 字符
- 段落作为基本单位

**示例**：
```
原文：
段落1：这是第一段内容，共200字符。
这还是第一段（单个换行不切分）。
⬅️ 双换行符在这里
段落2：这是第二段内容，共250字符。
⬅️ 双换行符在这里
段落3：这是第三段内容，共300字符。

切分结果：
块1：段落1（包含两行，因为只有单个换行）
块2：段落2 + 段落3（250 + 300 = 550字符，略超限制会智能处理）

💡 重点：只有双换行（\n\n）才会切分，单换行（\n）会保留
```

**优点**：
- ✅ 保持段落完整性
- ✅ 语义连贯性最好
- ✅ 适合结构化文档

**缺点**：
- ❌ 块大小不均匀
- ❌ 段落过长时仍需切分

**适用场景**：
- 📚 教材、课本
- 📖 小说、文章
- 📋 结构化文档
- 📝 Markdown文档

---

### 3️⃣ 按句子切分

**原理**：
- 以句号（。）、问号（？）、叹号（！）为句子分隔符
- 将多个句子合并成一个文本块，直到接近块大小限制

**参数**：
- 块大小限制：500 字符
- 句子作为基本单位

**示例**：
```
原文：
这是第一句话。这是第二句话？这是第三句话！这是第四句话。

切分结果：
块1：这是第一句话。这是第二句话？这是第三句话！
块2：这是第四句话。
```

**优点**：
- ✅ 句子完整，语义清晰
- ✅ 适合问答场景

**缺点**：
- ❌ 块大小差异大
- ❌ 可能丢失上下文

**适用场景**：
- ❓ FAQ（常见问题）
- 💬 问答对话
- 📌 知识点卡片
- 🔖 短文本集合

---

### 4️⃣ 自定义切分

**原理**：
- 手动设置块大小和重叠大小
- 按固定大小切分，但使用自定义参数

**参数**：
- 块大小：100-2000 字符（可调）
- 重叠大小：0-200 字符（可调）

**推荐参数**：
- 短文本：块大小 300，重叠 30
- 中等文本：块大小 500，重叠 50（默认）
- 长文本：块大小 800，重叠 80

**优点**：
- ✅ 灵活度最高
- ✅ 可针对特殊文档优化

**缺点**：
- ❌ 需要专业知识
- ❌ 调试成本高

**适用场景**：
- 🔬 特殊格式文档
- 🎯 性能优化
- 🧪 实验性场景

## 🔧 技术实现

### 后端API

**上传接口参数**：
```python
POST /api/kb-documents/{kb_uuid}

Parameters:
- file: UploadFile (必需)
- title: str (可选)
- auto_embedding: bool = True
- split_mode: str = "fixed"  # fixed/paragraph/sentence/custom
- chunk_size: int (仅自定义模式)
- chunk_overlap: int (仅自定义模式)
```

### 数据存储

切分配置保存在文档的 `meta_data` 字段中：

```json
{
  "split_mode": "paragraph",
  "chunk_size": null,
  "chunk_overlap": null
}
```

向量化时会读取这个配置进行相应的切分。

## ❓ 常见问题

### Q1: 如何知道选择哪种切分方式？

**A**: 对于大多数情况，选择**固定大小（默认）**即可。只有在以下情况考虑其他方式：
- 文档结构非常清晰（如教材）→ 按段落
- FAQ或问答文档 → 按句子
- 需要精确控制 → 自定义

### Q2: 块大小如何影响检索效果？

**A**: 
- **块太小**：
  - ✅ 检索更精确
  - ❌ 缺少上下文，响应质量下降
  
- **块太大**：
  - ✅ 上下文丰富
  - ❌ 检索不够精确，可能返回无关内容

- **推荐**：300-600 字符

### Q3: 重叠大小的作用是什么？

**A**: 重叠区域确保相邻文本块之间有共同内容，避免关键信息被切分到两个块的边界处。

**示例**：
```
无重叠：
块1: "...关键信息的前半部分"
块2: "后半部分..."  → 检索可能漏掉完整信息

有重叠：
块1: "...关键信息的前半部分和"
块2: "前半部分和后半部分..."  → 完整信息被保留
```

### Q4: 能否修改已上传文档的切分方式？

**A**: 目前不支持。如需更换切分方式，请：
1. 删除原文档
2. 重新上传并选择新的切分方式
3. 重新向量化

### Q5: 不同切分方式能否混用？

**A**: 可以。每个文档都独立保存切分配置，同一个知识库中的不同文档可以使用不同的切分方式。
