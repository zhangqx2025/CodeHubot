## 1. 项目背景

### 1.1 项目概述

CodeHubot 是一个**物联网+AI教学平台**，面向中小学校提供智能物联网设备管理和AI智能体教学服务。

### 1.2 核心场景

- **学校**：多个学校独立使用，数据完全隔离
- **班级**：每个班级有多个学习小组
- **设备**：教师将物联网设备分发给小组，学生在小组内共享使用
- **智能体**：学生创建AI智能体完成作业，教师可以查看和试用

### 1.3 目标用户

#### 两种用户类型 ⭐

**独立用户（Individual User）**
- **注册方式**：用户自主注册（默认）
- **特点**：不属于任何学校（`school_id = NULL`）
- **功能**：注册设备、使用智能体、个人学习
- **菜单**：精简版（无学校相关功能）

**机构用户（School User）**
- **平台管理员**：管理多个学校，分配学校管理员
- **学校管理员**：由平台管理员分配，管理本校的教师、学生、设备
- **教师**：由学校管理员创建，管理班级、分发设备、指导学生
- **学生**：由教师或学校管理员创建（支持批量导入），使用设备、创建智能体、完成实验

## 2. 核心需求

### 2.1 多租户架构

✅ **需求描述：**
- 支持多个学校独立使用
- 每个学校的数据完全隔离
- 学校管理员只能管理本校数据
- 平台管理员可以管理所有学校

### 2.2 设备分发管理

✅ **需求描述：**
- 一个设备可以分发给多个班级的小组
- 学生只能看到分发给本小组的设备
- 分发是持久化的，不需要每次上课重新分发
- 只有设备所有者（管理员/教师）可以管理设备

✅ **关键规则：**
- 一个小组固定使用特定的设备
- 学生不能看到其他小组的设备，避免误操作
- 设备所有权明确（属于学校或教师）

### 2.3 教师查看学生智能体

✅ **需求描述：**
- 教师能看到班级学生创建的智能体
- 教师可以直接使用学生的智能体（无需复杂的试用流程）
- 教师使用时，对话记录归属到教师自己
- 教师只能查看配置，不能编辑学生的智能体

### 2.4 权限隔离

✅ **需求描述：**
- 学生只能看到本小组的设备
- 教师只能管理自己班级的学生
- 学校管理员只能管理本校数据
- 平台管理员管理所有数据

## 3. 数据唯一性设计原则 ⭐⭐

### 3.1 唯一性规则详解

#### 全局唯一字段（跨所有学校）

| 字段 | 表 | 原因 | 示例 |
|------|-----|------|------|
| `username` | `aiot_core_users` | 登录凭证，必须全局唯一 | `john_doe` 只能有一个 |
| `email` | `aiot_core_users` | 联系方式，用于找回密码 | `teacher@example.com` 只能有一个 |
| `school_code` | `aiot_schools` | 学校识别码，用于登录 | `BJSY001` 只能有一个 |
| `device_uuid` | `aiot_core_devices` | 设备全局标识 | 每个设备唯一 |
| `agent_uuid` | `aiot_agents` | 智能体全局标识 | 每个智能体唯一 |

#### 学校内唯一字段（同一学校内唯一，不同学校可以重复）⭐

| 字段 | 表 | 唯一性约束 | 为什么可以重复 |
|------|-----|-----------|--------------|
| `teacher_number` | `aiot_core_users` | `(school_id, teacher_number)` | 每个学校有自己的工号规则 |
| `student_number` | `aiot_core_users` | `(school_id, student_number)` | 每个学校有自己的学号规则 |
| `class_name` | `aiot_classes` | `(school_id, class_name)` | 每个学校都有"三年级1班" |
| `group_name` | `aiot_groups` | `(class_id, group_name)` | 每个班级都有"小组A" |

#### 实际案例说明

**场景：两个学校都有工号 T001 的教师**

```yaml
学校A - 北京实验小学:
  教师:
    - 用户名: teacher_zhang  # 全局唯一 ✅
    - 工号: T001              # 学校内唯一 ✅
    - 学校: 北京实验小学 (ID=1)

学校B - 上海育才中学:
  教师:
    - 用户名: teacher_li     # 全局唯一（不同） ✅
    - 工号: T001              # 学校内唯一（可以和学校A重复） ✅
    - 学校: 上海育才中学 (ID=2)

结论: ✅ 两个教师可以同时存在，因为：
  - 用户名不同（全局唯一）
  - 工号虽然相同，但属于不同学校（学校内唯一）
  - (school_id=1, teacher_number='T001') ✅
  - (school_id=2, teacher_number='T001') ✅
```

**场景：同一学校内不能有重复工号**

```yaml
学校A - 北京实验小学:
  教师1:
    - 用户名: teacher_zhang
    - 工号: T001
    - 学校: 北京实验小学 (ID=1) ✅

  教师2（尝试创建）:
    - 用户名: teacher_wang
    - 工号: T001              # ❌ 与教师1冲突
    - 学校: 北京实验小学 (ID=1)
    
结论: ❌ 创建失败
  - 提示: "工号 T001 在本校已被使用"
  - 唯一性约束违反: (school_id=1, teacher_number='T001')
```

### 3.3 登录方式设计 ⭐⭐⭐（核心）

#### 方式一：普通登录（适合独立用户）

```yaml
登录界面:
  输入框1: 用户名 [john_doe_____]
  输入框2: 密码 [************]
  按钮: [登录]

适用用户:
  - ✅ 独立用户（必须使用）
  - ✅ 有普通用户名的机构用户（可选使用）

示例:
  用户名: john_doe
  密码: Password@123
  
验证逻辑:
  SELECT * FROM users 
  WHERE username = 'john_doe' 
  AND password_hash = hash('Password@123')
```

---

#### 方式二：机构登录（适合机构用户）⭐⭐

```yaml
登录界面:
  输入框1: 学校代码 [BJ-YCZX____]
  输入框2: 工号/学号 [T001________]
  输入框3: 密码 [************]
  按钮: [登录]

适用用户:
  - ✅ 教师（使用工号）
  - ✅ 学生（使用学号）
  - ✅ 学校管理员（使用工号）

示例1 - 教师登录:
  学校代码: BJ-YCZX
  工号: T001
  密码: Teacher@123

示例2 - 学生登录:
  学校代码: BJ-YCZX
  学号: 2025001
  密码: Student@123

验证逻辑:
  # 步骤1: 根据学校代码查找学校
  school = SELECT * FROM schools WHERE code = 'BJ-YCZX'
  
  # 步骤2: 在该学校内查找用户（工号或学号）
  user = SELECT * FROM users 
  WHERE school_id = school.id 
  AND (teacher_number = 'T001' OR student_number = '2025001')
  AND password_hash = hash('Teacher@123')
```

**优势：**
- ✅ 不需要记住复杂的用户名
- ✅ 使用熟悉的工号/学号
- ✅ 自动关联到学校
- ✅ 符合机构使用习惯

### 3.6 数据隔离设计原则 ⭐

#### 原则1: 数据访问权限

```yaml
独立用户（school_id = NULL）:
  可以访问:
    - ✅ 自己的设备（user_id = 当前用户）
    - ✅ 自己的智能体（user_id = 当前用户）
    - ✅ 自己的对话记录
  不能访问:
    - ❌ 任何学校的数据
    - ❌ 其他用户的数据

学生（school_id = 1）:
  可以访问:
    - ✅ 小组的设备（通过 device_distributions）
    - ✅ 自己的智能体
    - ✅ 小组信息
  不能访问:
    - ❌ 其他小组的设备
    - ❌ 其他学校的数据
    - ❌ 其他学生的智能体

教师（school_id = 1）:
  可以访问:
    - ✅ 自己的设备
    - ✅ 自己班级学生的设备
    - ✅ 自己班级学生的智能体（只读）
    - ✅ 本校的产品信息
  不能访问:
    - ❌ 其他班级的详细数据
    - ❌ 其他学校的数据

学校管理员（school_id = 1）:
  可以访问:
    - ✅ 本校所有用户
    - ✅ 本校所有设备
    - ✅ 本校所有班级和小组
    - ✅ 本校统计数据
  不能访问:
    - ❌ 其他学校的数据

平台管理员（school_id = NULL）:
  可以访问:
    - ✅ 所有学校的数据
    - ✅ 所有用户的数据
    - ✅ 全局统计
```

#### 原则2: 设备UUID重置机制 ⭐⭐⭐（重要安全特性）

**核心需求：** 教师可以随时重置设备的UUID，防止UUID泄露带来的安全风险

**为什么需要重置UUID？**

```yaml
场景1 - UUID意外泄露:
  问题: 学生或外部人员获取了设备UUID
  风险: 可能通过API直接访问或控制设备
  解决: 教师重置UUID，原UUID立即失效 ✅

场景2 - 设备转让或重新分配:
  问题: 设备从一个班级转到另一个班级
  需求: 清除旧的UUID历史记录
  解决: 重置UUID，相当于"重新注册"设备 ✅

场景3 - 安全审计:
  问题: 定期进行设备安全检查
  需求: 更新设备标识，提升安全性
  解决: 批量重置UUID ✅

场景4 - 防止学生绕过权限:
  问题: 学生记录了设备UUID，尝试直接调用API
```

```python
def reset_device_uuid(device_uuid, reason, operator):
    """重置设备UUID"""
    
    # 步骤1: 查找设备
    device = db.query(Device).filter(Device.uuid == device_uuid).first()
    if not device:
        raise NotFoundError("设备不存在")
    
    # 步骤2: 权限检查
    if not can_manage_device(operator, device):
        raise PermissionError("无权重置此设备的UUID")
    
    # 步骤3: 生成新UUID
    old_uuid = device.uuid
    new_uuid = generate_uuid()  # 保证全局唯一
    
    # 步骤4: 保存历史记录
    history = DeviceUUIDHistory(
        device_id=device.id,
        old_uuid=old_uuid,
        new_uuid=new_uuid,
        reset_by=operator.id,
        reset_reason=reason,
        ip_address=get_client_ip()
    )
    db.add(history)
    
    # 步骤5: 更新设备表
    device.uuid = new_uuid
    device.uuid_reset_count += 1
    device.last_uuid_reset_at = datetime.now()
    device.last_uuid_reset_by = operator.id
    
    # 添加到历史记录JSON
    if not device.previous_uuids:
        device.previous_uuids = []
    device.previous_uuids.append({
        'uuid': old_uuid,
        'reset_at': datetime.now().isoformat()
    })
    
    # 步骤6: 更新关联表
    # 设备分发记录中的device_uuid不需要更新（使用device_id外键）
    
    # 步骤7: 通知设备（MQTT）
    mqtt_publish(
        topic=f"devices/{old_uuid}/system",
        payload={
            "action": "uuid_updated",
            "new_uuid": new_uuid
        }
    )
    
    # 步骤8: 提交事务
    db.commit()
    
    # 步骤9: 记录操作日志
    log_operation(
        action="reset_device_uuid",
        device_id=device.id,
        old_uuid=old_uuid,
        new_uuid=new_uuid,
        operator=operator.id
    )
    
    return {
        'old_uuid': old_uuid,
        'new_uuid': new_uuid,
        'reset_at': datetime.now(),
        'reset_by': operator.real_name
    }
```

**旧UUID访问处理：**

```python
def get_device_by_uuid(uuid):
    """根据UUID获取设备（支持旧UUID查询）"""
    
    # 步骤1: 尝试用当前UUID查找
    device = db.query(Device).filter(Device.uuid == uuid).first()
    if device:
        return device, True  # 返回设备和有效标志
    
    # 步骤2: 检查是否是旧UUID
    history = db.query(DeviceUUIDHistory).filter(
        DeviceUUIDHistory.old_uuid == uuid
    ).order_by(DeviceUUIDHistory.reset_at.desc()).first()
    
    if history:
        # 找到对应的设备（通过device_id）
        device = db.query(Device).filter(Device.id == history.device_id).first()
        if device:
            return device, False, history  # 返回设备、无效标志、历史记录
    
    # 步骤3: 完全找不到
    return None, False, None

# API使用
@router.get("/devices/{device_uuid}")
def get_device(device_uuid: str):
    device, is_valid, history = get_device_by_uuid(device_uuid)
    
    if not device:
        raise HTTPException(404, "设备不存在")
    
    if not is_valid:
        # 旧UUID访问
        return {
            "code": 410,  # Gone
            "message": "此UUID已失效",
            "data": {
                "old_uuid": device_uuid,
                "new_uuid": device.uuid,
                "reset_at": history.reset_at,
                "reset_by": history.reset_by,
                "reason": history.reset_reason
            }
        }
    
    # 正常返回设备信息
    return {
        "code": 200,
        "data": device
    }
```

```python
            return query  # 不过滤
```

### 9.2 权限验证装饰器

```python
from functools import wraps

def require_role(*allowed_roles):
    """角色权限装饰器"""
    def decorator(func):
        @wraps(func)
        async def wrapper(*args, current_user: User = Depends(get_current_user), **kwargs):
            if current_user.role not in allowed_roles:
                raise HTTPException(403, "权限不足")
            return await func(*args, current_user=current_user, **kwargs)
        return wrapper
    return decorator

# 使用示例
@router.post("/schools")
@require_role('platform_admin')
```

### 9.3 设备访问验证

```python
def verify_device_access(device_uuid: str, user: User, db: Session):
    """验证用户是否可以访问设备"""
    device = db.query(Device).filter(Device.uuid == device_uuid).first()
    
    # 1. 设备所有者
    if device.user_id == user.id:
        return True
    
    # 2. 学生：检查是否分发给其小组
    if user.role == 'student':
        group_member = db.query(GroupMember).filter(
            GroupMember.student_id == user.id
        ).first()
        
        if not group_member:
            raise HTTPException(403, "您不属于任何小组")
        
        distribution = db.query(DeviceDistribution).filter(
            DeviceDistribution.device_uuid == device_uuid,
            DeviceDistribution.group_id == group_member.group_id,
            DeviceDistribution.status == 'active'
        ).first()
        
        if not distribution:
            raise HTTPException(403, "该设备未分发给您的小组")
        
        return True
    
    # 3. 教师：检查是否是自己分发的设备
    elif user.role == 'teacher':
        if device.user_id == user.id:  # 自己的设备
            return True
        
        # 查看是否分发过（可以查看使用情况）
        distribution = db.query(DeviceDistribution).filter(
            DeviceDistribution.device_uuid == device_uuid,
            DeviceDistribution.distributor_id == user.id
        ).first()
        
        if distribution:
            return True  # 只读权限
        
        raise HTTPException(403, "无权访问该设备")
    
    # 4. 管理员
    elif user.role in ['school_admin', 'platform_admin']:
        return True
    
    raise HTTPException(403, "无权访问")
```

### 15.4 设备分时分配机制 ⭐⭐⭐（多班级共享安全）V2.0

**核心问题：** 一个设备被分配给多个班级，如何防止一个班级正在做实验时，其他班级学生捣乱？

#### 解决方案：设备分时分配 + 按时段可见 ⭐⭐⭐

**核心理念：** 学生只在上课时才能看到设备

**设计思路：**

```yaml
场景 - 三年级1班周一下午的物理课:
  
  提前分配（周日）:
    - 张老师分配设备给1班
    - 时间段：每周一 14:00-14:45
    - 设备对学生不可见 ✅
  
  上课前（14:00前）:
    - 1班学生：看不到设备 ✅
    - 2班学生：看不到设备 ✅
    - 3班学生：看不到设备 ✅
  
  开始上课（14:00）⭐:
    - 教师点击"开始上课"
    - 设备对1班学生可见 ✅
    - 1班学生可以看到和使用
    - 2班、3班学生：完全看不到设备 ✅
  
  上课中（14:00-14:45）:
    - 1班学生：正常使用设备 ✅
    - 2班、3班：完全看不到，无法干扰 ✅
  
  结束上课（14:45）⭐:
    - 教师点击"结束上课"或自动结束
    - 设备对1班学生隐藏 ✅
    - 所有学生都看不到设备
  
  下周重复:
    - 下周一14:00，教师再次点击"开始上课"
    - 设备再次对1班可见 ✅
```

**核心优势：**

```yaml
对比旧方案:
  旧方案（锁定模式）:
    - 设备始终对所有班级可见
    - 显示"使用中"状态
    - 其他班级可以查看（只读）
    问题: 复杂、容易困惑、仍可能干扰
  
  新方案（可见性控制）⭐:
    - 设备只在上课时对当前班级可见
    - 其他班级完全看不到
    - 天然隔离，物理上无法干扰
    优势: 简洁、清晰、稳定、安全
```

#### 权限验证逻辑

```python
def can_access_device(user: User, device: Device, db: Session) -> tuple[bool, str]:
    """
    检查用户是否可以访问设备
    
    返回: (是否允许, 拒绝原因)
    """
    
    # 步骤1: 管理员始终可以访问
    if user.role in ['platform_admin', 'school_admin']:
        return True, ""
    
    # 步骤2: 检查设备是否有活跃的上课会话
    active_session = db.query(DeviceSession).filter(
        DeviceSession.device_id == device.id,
        DeviceSession.status == 'active'
    ).first()
    
    if not active_session:
        # 没有活跃会话，检查基本分发权限
        return has_distribution_access(user, device, db)
    
    # 步骤3: 有活跃会话，检查用户是否属于占用班级/小组
    if active_session.group_id:
        # 锁定到小组级别
        is_in_group = db.query(GroupMember).filter(
            GroupMember.group_id == active_session.group_id,
            GroupMember.student_id == user.id
        ).count() > 0
        
        if is_in_group:
            return True, ""
        else:
            return False, f"设备正在被{active_session.class_name}-{active_session.group_name}使用"
    
    else:
        # 锁定到班级级别
        is_in_class = db.query(ClassStudent).filter(
            ClassStudent.class_id == active_session.class_id,
            ClassStudent.student_id == user.id
        ).count() > 0
        
        if is_in_class:
            return True, ""
        else:
            return False, f"设备正在被{active_session.class_name}使用"


def has_distribution_access(user: User, device: Device, db: Session) -> tuple[bool, str]:
    """检查用户是否有设备分发权限"""
    
    # 查找用户所在的小组
    user_groups = db.query(GroupMember.group_id).filter(
        GroupMember.student_id == user.id
    ).all()
    
    group_ids = [g[0] for g in user_groups]
    
    # 检查设备是否分发给这些小组
    distribution = db.query(DeviceDistribution).filter(
        DeviceDistribution.device_id == device.id,
        DeviceDistribution.group_id.in_(group_ids)
    ).first()
    
    if distribution:
        return True, ""
    else:
        return False, "您没有权限访问此设备"
```

```python
# 定时任务（每分钟执行）
@scheduler.task('interval', minutes=1)
def auto_unlock_expired_sessions():
    """自动解锁超时的会话"""
    
    now = datetime.now()
    
    # 查找所有应该自动解锁的会话
    expired_sessions = db.query(DeviceSession).filter(
        DeviceSession.status == 'active',
        DeviceSession.scheduled_end_at < now
    ).all()
    
    for session in expired_sessions:
        # 结束会话
        session.status = 'ended'
        session.actual_end_at = now
        
        # 记录日志
        log_info(f"设备 {session.device_id} 自动解锁（预定时间到期）")
        
        # 发送通知给教师
        notify_teacher(
            session.teacher_id,
            f"设备 {session.device.device_name} 已自动解锁"
        )
    
    db.commit()
```

```python
@scheduler.task('interval', minutes=5)
def auto_unlock_inactive_sessions():
    """自动解锁长时间无操作的会话"""
    
    now = datetime.now()
    inactive_threshold = now - timedelta(minutes=15)  # 15分钟无操作
    
    # 查找长时间无操作的会话
    inactive_sessions = db.query(DeviceSession).filter(
        DeviceSession.status == 'active',
        DeviceSession.last_activity_at < inactive_threshold
    ).all()
    
    for session in inactive_sessions:
        # 结束会话
        session.status = 'ended'
        session.actual_end_at = now
        
        # 记录日志
        log_warning(f"设备 {session.device_id} 自动解锁（15分钟无操作）")
        
        # 发送通知给教师
        notify_teacher(
            session.teacher_id,
            f"设备 {session.device.device_name} 因长时间无操作已自动解锁"
        )
    
    db.commit()
```

### 15.5 UUID可见性控制 ⭐⭐⭐（灵活安全策略）

**核心功能：** 教师可以设置学生是否可以看到设备UUID

**设计理念：**

```yaml
不同教学场景的需求:

场景A - 安全优先（默认）⭐:
  需求: 隐藏UUID，防止泄露
  示例: 常规实验课（95%场景）
  配置: show_uuid_to_students = FALSE ✅
  学生看到: 
    - 设备名称 ✅
    - 设备ID ✅
    - UUID ❌（不显示）

场景B - 教学需要:
  需求: 显示UUID，学习设备标识概念
  示例: 物联网概念课、设备管理课
  配置: show_uuid_to_students = TRUE
  学生看到:
    - 设备名称 ✅
    - 设备ID ✅
    - UUID ✅（显示）
    - [复制UUID]按钮

场景C - 临时调试:
  需求: 临时显示UUID排查问题
  示例: 连接故障诊断
  流程:
    - 教师开启显示
    - 学生提供UUID
    - 问题解决
    - 教师关闭显示
```

**数据库设计：**

```sql
ALTER TABLE `aiot_device_distributions`
ADD COLUMN `show_uuid_to_students` BOOLEAN DEFAULT FALSE COMMENT '是否向学生显示UUID';
```

**API接口：**

```yaml
接口: PUT /api/distributions/{uuid}/uuid-visibility
权限: 教师、管理员

请求:
  {
    "show_uuid_to_students": true,
    "reason": "物联网概念学习"
  }

响应:
  {
    "code": 200,
    "message": "UUID可见性已更新",
    "data": {
      "show_uuid_to_students": true
    }
  }
```

**学生API响应示例：**

```json
// UUID隐藏（默认）
{
  "device_id": "AIOT-ESP32-81417E8A",
  "device_name": "温湿度传感器",
  // 不返回uuid字段
  "uuid_visible": false
}

// UUID显示（教师开启后）
{
  "uuid": "2b91058a-dfa7-42f1-b084-3cddd4edf788",  // ⭐
  "device_id": "AIOT-ESP32-81417E8A",
  "device_name": "温湿度传感器",
  "uuid_visible": true
}
```

**多层安全防护配合：**

```yaml
安全特性组合使用:

特性1 - UUID可见性控制:
  作用: 默认隐藏UUID
  配置: 教师可灵活开启/关闭
  
特性2 - 设备分时可见:
  作用: 学生只在上课时看到设备
  配置: 基于时间段自动管理
  
特性3 - UUID重置:
  作用: UUID泄露后快速重置
  配置: 教师一键重置

组合效果:
  即使教师开启了UUID显示，
  学生也只能在上课时看到，
  课程结束后立即隐藏，
  即使泄露也可以快速重置 ✅
  
  三重保护，安全可靠 ⭐⭐⭐
```

### 15.6 UUID重置机制详解 ⭐⭐⭐（核心安全特性）

**应用场景：**

```yaml
场景1 - UUID泄露应急:
  触发: 发现UUID被学生记录或泄露
  操作: 教师立即重置UUID
  效果: 旧UUID立即失效，所有使用旧UUID的访问被阻止
  时间: < 1分钟完成

场景2 - 定期安全轮换:
  触发: 每学期开始
  操作: 批量重置所有设备UUID
  效果: 提升整体安全性，清除历史访问记录
  时间: 批量操作，约5分钟

场景3 - 设备转让:
  触发: 设备从一个班级转到另一个班级
  操作: 重置UUID + 解除旧分发 + 创建新分发
  效果: 旧班级无法继续访问，新班级获得全新UUID
  时间: < 2分钟完成

场景4 - 异常访问检测:
  触发: 监控系统发现异常访问模式
  操作: 自动通知教师，教师决定是否重置
  效果: 快速响应安全威胁
  时间: 教师收到通知后5分钟内处理
```

**安全保障：**

```yaml
技术保障:
  - ✅ 操作不可逆（旧UUID永久失效）
  - ✅ 全链路更新（数据库 + MQTT + 前端）
  - ✅ 历史记录完整（audit log）
  - ✅ 旧UUID访问返回410 Gone

管理保障:
  - ✅ 权限严格控制（只有设备所有者）
  - ✅ 频率限制（24小时内最多3次）
  - ✅ 操作日志记录（谁、何时、为何）
  - ✅ 通知机制（相关教师自动通知）

监控保障:
  - ✅ UUID重置统计（异常检测）
  - ✅ 访问模式分析（发现异常）
  - ✅ 实时告警（高频重置提醒）
```
